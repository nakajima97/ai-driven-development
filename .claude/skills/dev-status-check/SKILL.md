## 概要

プロジェクトの技術的ベースラインを自動収集・整理する。後続のすべてのskillがこの結果を参照するため、正確かつ網羅的に情報を収集することが重要。

## 対象範囲

フロントエンド・バックエンド・インフラ・テストコードのすべてを対象とする。

## 手順

### 1. 基本情報の収集

以下をすべて確認する:

- `package.json` / `Gemfile` / `pyproject.toml` / `go.mod` など依存関係ファイル
- `README.md` / `CLAUDE.md` など既存ドキュメント
- ディレクトリ構造（主要なフォルダと役割）
- 設定ファイル（`.eslintrc`, `tsconfig.json`, `vite.config.ts` など）
- CI/CD 設定（`.github/workflows/` など）
- テスト設定とテストランナー

### 2. 技術スタックの判定

収集した情報から以下を特定する:

- **言語・フレームワーク**: バージョンも含めて記録
- **フロントエンド技術**: React/Vue/Svelte など、スタイリング手法（Tailwind/CSS Modules等）
- **バックエンド技術**: Node.js/Rails/FastAPI など
- **データベース**: 種類と ORM/クライアント
- **テストツール**: Jest/Vitest/Playwright/RSpec など
- **ビルドツール・パッケージマネージャー**

### 3. 既存パターンの抽出

コードを読んで以下を理解する:

- ディレクトリ構成の規則（`src/components/`, `src/pages/` など）
- 命名規則（キャメルケース/スネークケースなど）
- コンポーネント/モジュールの設計パターン
- エラーハンドリングの方針
- 状態管理の方法（Redux/Zustand/Context など）

### 4. 未定義項目の確認

アーキテクチャや規約が決まっていない場合は、ユーザーに定義を促してワークフローを止める。
確認が必要な項目は GitHub Issue にコメントとして記録する。

## 成果物

### 必須（Issue コメントに記録）

現状確認レポートとして以下をまとめる:

```
## 現状確認レポート

### 技術スタック
- フロントエンド: [フレームワーク + バージョン]
- バックエンド: [フレームワーク + バージョン]
- DB: [種類 + ORM]
- テスト: [ツール一覧]
- ビルド/パッケージ: [ツール一覧]

### ディレクトリ構造
[主要ディレクトリと役割]

### コーディング規約
- 命名規則: [ルール]
- ファイル構成: [パターン]
- その他: [注意点]

### 確認が必要な項目
- [未定義の項目があれば列挙]
```

### 状況に応じて（Markdown に記録）

標準化ルール（プロジェクト全体で採用すべき規約が見つかった場合）を `docs/` 配下に記録する。

## 次のステップ

現状確認が完了したら、「2. 要件定義 Skill (`dev-requirements`)」への移行を提案する。

## 概要

リファクタリングまで完了した成果を PR として共有し、品質保証を進める。PRの本文に変更の目的・影響範囲・確認手順を明示することで、レビュワーの負荷を下げる。

## 前提条件

`dev-refactoring` が完了していること。未完了の場合はエラーとして処理し、先にそちらを実行するよう促す。

## 事前確認

### CI の状態確認

PR 作成前に以下が通ることを確認する（`dev-status-check` の結果による）:

- テストがすべてグリーン
- lint / format / 型チェックが通る
- ビルドが成功する

### ブランチ名の確認

プロジェクトのブランチ命名規則に従っているか確認する。一般的な規則:
- `feature/[機能名]`
- `fix/[バグ名]`
- `chore/[作業内容]`

## 手順

### 1. ブランチの push

```bash
git push -u origin [ブランチ名]
```

### 2. PR 本文の作成

`dev-status-check` で確認したPRテンプレートに従う。テンプレートがない場合は以下の構成で作成:

```markdown
## 概要
[何をするPRか、1〜2文で]

## 変更の目的
[なぜこの変更が必要か]

## 変更内容
- [変更点1]
- [変更点2]

## 影響範囲
- [影響するファイル・機能]
- [影響しない範囲]

## 確認手順
1. [ブランチをチェックアウト]
2. [XXXページを開く]
3. [YYYの操作をする]
4. [ZZZが表示されることを確認]

## テスト
- [ ] ユニットテスト: パス
- [ ] 統合テスト: パス
- [ ] E2Eテスト: パス（対象がある場合）
- [ ] 手動確認: 完了

## スクリーンショット
[UI変更がある場合は Before / After を添付]
```

### 3. レビュー依頼の準備

レビューコメントへの対応方針:
- 全コメントに返信する
- 対応した修正は「対応しました」とコメントする
- 対応しない場合は理由を説明する
- レビューコメントへの対応は PR の conversation で管理する

### 4. PR の作成

```bash
gh pr create --title "[タイトル]" --body "[本文]"
```

## 成果物

### 必須（PR に記録）

- PR リンク
- レビュー依頼メッセージ

## レビューコメント対応

レビュー後の修正:
1. フィードバックに基づいてコードを修正
2. 追加コミットを push（squash はマージ時に行う）
3. PR の conversation でコメントに返信
4. 再レビューを依頼

## ワークフロー完了

PR がマージされたらワークフロー完了。必要に応じて:
- 関連 Issue を close する
- 次の機能開発のために「1. 現状確認 (`dev-status-check`)」から再スタート

# 要件定義: 古いログデータ夜間バッチ削除処理

## 1. 概要

バックグラウンドで夜間に古いログデータを自動削除するバッチ処理を追加する。
システムのストレージ肥大化を防ぎ、パフォーマンスを維持することを目的とする。

## 2. ユーザーストーリー

- システム管理者として、古いログデータが自動的に削除されることで、手動メンテナンス作業を削減したい
- 開発者として、ログデータの保持期間を設定できることで、コンプライアンス要件に対応したい
- 運用担当者として、バッチ処理の実行結果を確認できることで、正常に動作しているか把握したい

## 3. 機能要件

### 3.1 バッチ処理本体

| # | 要件 | 優先度 |
|---|------|--------|
| F-01 | 指定した保持期間（日数）より古いログデータを削除する | 必須 |
| F-02 | 夜間（デフォルト: 毎日 02:00 JST）に自動実行される | 必須 |
| F-03 | 削除対象の件数をドライラン（dry-run）で事前確認できる | 必須 |
| F-04 | 削除処理はバッチサイズ単位で分割して実行し、DBへの負荷を抑える | 必須 |
| F-05 | 処理中にエラーが発生した場合、ロールバックせずエラーログを記録して続行する | 必須 |
| F-06 | 処理の開始・終了・削除件数・エラー内容をログに出力する | 必須 |
| F-07 | 実行中の多重起動を防止するロック機構を持つ | 必須 |
| F-08 | 手動でバッチ処理を実行できる管理用APIエンドポイントを提供する | 推奨 |
| F-09 | 削除対象のログテーブル・対象カラムを設定ファイルで管理する | 推奨 |

### 3.2 設定項目

| 設定キー | デフォルト値 | 説明 |
|----------|-------------|------|
| `LOG_RETENTION_DAYS` | `90` | ログの保持日数 |
| `BATCH_CLEANUP_SCHEDULE` | `0 2 * * *` | cron形式のスケジュール |
| `BATCH_SIZE` | `1000` | 1回の削除件数 |
| `BATCH_DRY_RUN` | `false` | ドライランモード |

### 3.3 管理API（推奨）

- `POST /api/admin/batch/log-cleanup` — バッチ処理を手動実行
- `GET  /api/admin/batch/log-cleanup/status` — 直近の実行結果を取得

## 4. 非機能要件

| # | 要件 |
|---|------|
| NF-01 | バッチ処理は夜間の低トラフィック時間帯（00:00〜05:00 JST）に完了すること |
| NF-02 | 1回の実行で最大100万件削除できること（バッチ分割により対応） |
| NF-03 | バッチ処理がユーザー向けAPIのレスポンスタイムに影響しないこと |
| NF-04 | 削除処理は冪等性を持つこと（再実行しても同じ結果になること） |
| NF-05 | 本番環境での初回実行前に必ずドライランで件数を確認すること |

## 5. 技術設計方針

### 5.1 技術スタック

- **フレームワーク**: Next.js 14+ (App Router)
- **言語**: TypeScript
- **スタイリング**: Tailwind CSS（管理画面を追加する場合）
- **スケジューラ**: `node-cron` または Vercel Cron Jobs（デプロイ環境による）

### 5.2 実装アーキテクチャ

```
/app
  /api
    /admin
      /batch
        /log-cleanup
          route.ts        ← 手動実行API
          /status
            route.ts      ← 実行状態確認API

/lib
  /batch
    log-cleanup.ts        ← バッチ処理ロジック（コア）
    scheduler.ts          ← cronスケジューラ登録
    lock.ts               ← 多重起動防止ロック
    types.ts              ← 型定義

/config
  batch.ts                ← 保持日数・バッチサイズ等の設定
```

### 5.3 デプロイ環境別のスケジューラ選択

| 環境 | 推奨スケジューラ |
|------|----------------|
| Vercel | Vercel Cron Jobs (`vercel.json` で設定) |
| Node.jsサーバー / Docker | `node-cron` をサーバー起動時に登録 |
| AWS / GCP | Lambda / Cloud Functions + EventBridge / Cloud Scheduler |

### 5.4 Vercel環境の場合の設定例

`vercel.json` に以下を追加:
```json
{
  "crons": [
    {
      "path": "/api/admin/batch/log-cleanup",
      "schedule": "0 17 * * *"
    }
  ]
}
```
※ Vercel CronはUTC基準のため、JST 02:00 = UTC 17:00（前日）

## 6. データ設計

### 6.1 削除対象テーブルの前提条件

削除対象テーブルには以下のカラムが必要:
- `created_at: DateTime` — レコード作成日時（削除条件に使用）

### 6.2 バッチ実行履歴テーブル（新規追加）

```sql
CREATE TABLE batch_execution_logs (
  id           SERIAL PRIMARY KEY,
  batch_name   VARCHAR(100) NOT NULL,
  started_at   TIMESTAMP    NOT NULL,
  finished_at  TIMESTAMP,
  status       VARCHAR(20)  NOT NULL,  -- 'running' | 'completed' | 'failed'
  deleted_count INT         DEFAULT 0,
  error_message TEXT,
  dry_run      BOOLEAN      DEFAULT FALSE,
  created_at   TIMESTAMP    DEFAULT NOW()
);
```

## 7. エラーハンドリング

| エラーケース | 対応方針 |
|------------|---------|
| DB接続エラー | エラーログ記録後、処理を中断しアラート通知 |
| タイムアウト | バッチサイズを縮小して再試行、上限回数超過でエラー終了 |
| 多重起動検知 | 後発のプロセスを即時終了（エラーにはしない） |
| 削除件数が閾値超過 | 警告ログを出力し処理は継続（異常データ検知用） |

## 8. セキュリティ要件

- 手動実行APIは管理者ロールのみアクセス可能（認証ミドルウェアで制御）
- ドライランAPIは読み取り専用として権限を分離できる設計にする
- バッチ処理のログに個人情報を含めない

## 9. 受け入れ条件

- [ ] 保持期間（例: 90日）より古いレコードが削除される
- [ ] 保持期間内のレコードは削除されない
- [ ] ドライランモードでは実際の削除が行われない
- [ ] バッチ実行履歴がDBに記録される
- [ ] 多重実行時、2つ目のプロセスが安全に終了する
- [ ] 夜間スケジュールで自動実行される（ステージング環境で確認）
- [ ] 手動APIから実行できる
- [ ] エラー発生時にログが残り、バッチ履歴に `failed` が記録される

## 概要

実装計画・画面仕様をもとに、画面とロジックを守るためのテストを作成する。テスト失敗 = PRマージ不可を前提とし、クリティカルパスを優先してテストケースを設計する。

## 前提条件

`dev-implementation-plan` の実装計画ドキュメントが完了していること。未完了の場合はエラーとして処理し、先にそちらを実行するよう促す。

## テスト環境の確認

`dev-status-check` の結果から以下を確認する:

- テストフレームワーク（Jest / Vitest / RSpec / pytest 等）
- カバレッジツール（istanbul / nyc / simplecov 等）
- E2E テストツール（Playwright / Cypress 等）
- CI との連携方法

## テストの種類と優先度

### ユニットテスト（優先度: 高）

- 純粋関数・ユーティリティのテスト
- バリデーションロジック
- データ変換・フォーマット処理
- 状態管理（Redux / Zustand の reducer / action）

### 統合テスト（優先度: 中）

- コンポーネントの描画と操作
- API クライアントとのやり取り
- フォーム送信フロー

### E2E テスト（優先度: 中）

- クリティカルパス（ログイン→主要操作→完了）
- エラーフロー（入力ミス、API エラー）
- 主要なユーザーシナリオ

## 手順

### 1. テスト対象の特定

実装計画のタスク一覧から、テストが必要な箇所を特定する:

- 複雑なロジックを含む関数
- API 呼び出しを含むコンポーネント
- 状態変化を伴うインタラクション
- エラーハンドリング

### 2. テストケースの設計

各テスト対象に対して以下のシナリオを設計する:

- 正常系（期待通りの入力と出力）
- 異常系（不正な入力、エラー状態）
- 境界値（空文字、最大文字数、nullなど）
- 副作用の確認（API呼び出し回数など）

### 3. テストコードの実装

プロジェクトの既存テストパターンに従って実装する:

```typescript
// 例: ユニットテスト
describe('[対象機能]', () => {
  it('[正常系: 期待する動作]', () => {
    // Arrange
    // Act
    // Assert
  });

  it('[異常系: エラーケース]', () => {
    // ...
  });
});
```

### 4. テスト失敗条件の文書化

意図的にテストが失敗するケースと、その修正方針を記録する。

## 成果物

### 必須（コードに記録）

テストプランとテストケース一覧（ユニット/統合/E2E）をテストファイルとして実装:

- `*.test.ts` / `*.spec.ts` ファイル群
- E2E テストスペック（`e2e/` または `tests/` ディレクトリ）

### 状況に応じて（Issue コメントに記録）

失敗条件・修正案ドキュメント（テストが通らない既知の問題がある場合）

## 次のステップ

テストが作成されたら「6. コード実装 (`dev-code-implementation`)」への移行を提案する。
テストはこの時点では失敗して構わない（TDD アプローチ）。

## 概要

画面要件定義をもとに、実際のルートに静的ページを作成して要件を詰める。この段階ではバックエンド連携は行わず、画面のレイアウト・UX・インタラクションをユーザーと確認することが目的。

## 前提条件

`dev-requirements` の画面要件定義書が完了していること。未完了の場合はエラーとして処理し、先にそちらを実行するよう促す。

## 技術スタックの確認

`dev-status-check` の結果から以下を確認する:

- フロントエンドフレームワーク（React / Vue / Svelte / HTML 等）
- スタイリング手法（Tailwind CSS / CSS Modules / styled-components 等）
- Storybook の有無
- ルーティング方法（Next.js App Router / React Router 等）

出力形式はこれらに合わせる。

## 手順

### 1. 静的ページの作成場所を決定

要件定義で決まった実際のルートに静的画面を直接作成する。例:
- `/profile/{id}` にプロフィール画面を作成する場合 → そのルートに静的ページを実装
- `/dashboard` にダッシュボードを作成する場合 → そのルートに静的ページを実装

モックデータ（ハードコードされたダミーデータ）を使って表示を完成させる。

### 2. コンポーネント構造の設計

- ページコンポーネントの分割方針
- 再利用可能なコンポーネントの特定
- フォルダ構成（プロジェクトの既存パターンに従う）

### 3. 実装

以下を含めて実装する:

- レイアウト・ヘッダー・フッター
- 主要なコンテンツ領域
- フォームと入力要素（バリデーション UI も含む）
- ローディング状態・空状態・エラー状態
- レスポンシブ対応（モバイル・タブレット・デスクトップ）
- インタラクション（ホバー、フォーカス、アクティブ状態）

### 4. フィードバック収集

静的ページが動作する状態でユーザーに確認を求める:

- 画面の構成・レイアウトの確認
- 不足している要素・変更が必要な箇所
- 追加のインタラクション要件

フィードバックを反映してから次のステップに進む。

## 成果物

### 必須（コードに記録）

- 静的画面のコンポーネントファイル群
- モックデータファイル（`mocks/` または `fixtures/` ディレクトリ）

### 状況に応じて（Issue コメントに記録）

レスポンシブ対応チェックリスト:

```
- [ ] モバイル (320px-767px)
- [ ] タブレット (768px-1023px)
- [ ] デスクトップ (1024px+)
- [ ] 大画面 (1440px+)
```

## 次のステップ

ユーザーが静的画面を承認したら「4. 実装計画の立案 (`dev-implementation-plan`)」への移行を提案する。

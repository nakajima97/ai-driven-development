## 概要

完了したコードの品質を向上させる。スコープは**今回の変更箇所のみ**に限定し、無関係な既存コードには手を加えない。テストがグリーンな状態を維持しながら改善する。

## 前提条件

`dev-code-implementation` の実装が完了し、テストがすべてグリーンであること。未完了の場合はエラーとして処理し、先にそちらを実行するよう促す。

## リファクタリングの範囲

今回追加・変更したコードのみを対象とする。既存の無関係なコードへの変更は行わない（スコープクリープを防ぐため）。

## 技術負債の検出

`dev-status-check` で確認した静的解析ルール・レビューフィードバック・実行結果に基づいて以下を確認する:

- **コードの重複**: 同じロジックが複数箇所に存在
- **長い関数/コンポーネント**: 単一責任原則に違反
- **深いネスト**: 複雑な条件分岐
- **不明瞭な命名**: 意図が伝わらない変数名・関数名
- **未使用のコード**: 不要な import、変数、関数
- **パフォーマンス問題**: 不必要な再レンダリング、N+1 クエリなど

## 手順

### 1. 現状の分析

実装したコードを見直して改善点を特定する:

- 静的解析ツールの警告・エラーを確認
- テストカバレッジレポートの確認（カバーされていない箇所）
- コードレビューフィードバックの確認（あれば）

### 2. 改善提案の作成

特定した問題点を優先度付きでリスト化し、ユーザーに提案する:

```
## リファクタリング提案

### 優先度: 高
- [問題]: [現状の説明]
  [改善案]: [変更方針]

### 優先度: 中
- ...

### 優先度: 低（時間があれば）
- ...
```

### 3. ユーザーの承認後に実施

提案した改善案をユーザーが承認してから実施する。

### 4. テストの再確認

リファクタリング後もテストがグリーンであることを確認する。

### 5. モジュール分割・抽象化

重複コードがある場合は共通関数・フックに切り出す。ただし、1箇所でしか使われないものは無理に抽象化しない（YAGNI原則）。

## 成果物

### 必須（Issue コメントに記録）

リファクタリング提案書:

```
## リファクタリング提案書

### 実施した改善
- [改善内容と理由]

### 保留した改善（今回のスコープ外）
- [今後の対応が必要な項目]
```

### 状況に応じて

- コード改善差分パッチ（Issue コメント / PR に記録）
- ドキュメント更新内容（Markdown に記録）

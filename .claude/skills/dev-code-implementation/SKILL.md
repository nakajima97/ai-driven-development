## 概要

実装計画・テスト仕様・静的画面をもとに、テストをパスするコードを実装する。1タスク1コミットを単位として逐次進め、CI が通る状態を常に維持する。

## 前提条件

`dev-test-creation` のテストコードが完了していること。未完了の場合はエラーとして処理し、先にそちらを実行するよう促す。

## 実装方針

- **テストグリーンを目標**: 実装はテストをパスさせることを第一目標とする
- **1タスク1コミット**: 実装計画のタスク単位でコミットする
- **既存パターンに従う**: `dev-status-check` で確認したコードパターン・命名規則を遵守する
- **小さく始める**: 最初に動く最小実装を作り、テストを通してから改善する

## 手順

### 1. 実装の開始前確認

- 実装計画のタスク一覧を確認
- 対象タスクの依存関係を確認（前提タスクが完了しているか）
- 既存コードの関連部分を読んでパターンを把握

### 2. タスク単位で実装

各タスクを以下の手順で進める:

1. **実装**: 対象機能のコードを書く
2. **テスト実行**: 対象のテストが通るか確認
3. **CI 確認**: lint / format / 型チェックが通るか確認
4. **コミット候補の提示**: ユーザーにコミット内容を提示

### 3. コミットメッセージの形式

プロジェクトの既存コミットメッセージ形式に従う。`dev-status-check` で確認した慣例を使用。
一般的には:

```
[type]: [変更の概要]

- [詳細1]
- [詳細2]
```

### 4. フロントエンド実装のポイント

静的画面のモックデータを実際の API 呼び出しに置き換える:

- API クライアントの呼び出し実装
- ローディング状態の管理
- エラーハンドリングの実装
- 型定義の整備

### 5. バックエンド実装のポイント

- API エンドポイントの実装
- バリデーション処理
- DB 操作（実装計画のスキーマに従う）
- エラーレスポンスの統一

### 6. CI 準備

コミット前に確認:

- `dev-status-check` で確認した lint/format コマンドを実行
- 型エラーがないか確認
- テストがすべてパスするか確認

## 成果物

### 必須（コードに記録）

実装差分パッチとコード変更（リポジトリにコミット）

### 状況に応じて（PR に記録）

- 実装が完了したらPRドラフトを作成（必要に応じて）

## 次のステップ

すべてのタスクが実装完了・テストグリーンになったら「7. リファクタリング (`dev-refactoring`)」への移行を提案する。

## 概要

画面要件を構造化して整理する。API定義は行わず、画面の目的・構成・機能にフォーカスする。ユーザーとの認識合わせが主な目的。

## 前提条件

`dev-status-check` の現状確認レポートが完了していること。未完了の場合はエラーとして処理し、先にそちらを実行するよう促す。

## 手順

### 1. 入力情報の収集

以下のすべての情報源を確認する:

- GitHub Issue・PR 本文
- 既存の仕様書・設計文書
- ユーザーストーリー
- 既存 UI 要素（現行画面がある場合）
- 非機能要件（パフォーマンス、アクセシビリティなど）

### 2. 画面の要否判断

実装したい機能から「画面が必要か」を判定する:

- **画面が必要**: ユーザーが操作・閲覧するインターフェースが必要
- **画面が不要**: バッチ処理、API エンドポイント、バックグラウンド処理など

画面が不要な場合は「4. 実装計画の立案 (`dev-implementation-plan`)」へ移行を提案する。

### 3. 画面要件の整理

画面が必要と判断した場合、以下を構造化する:

**画面の目的と利用シナリオ**
- 誰が使うか（ユーザーペルソナ）
- どんな場面で使うか
- 達成したいゴール

**画面構成案**
- 画面一覧と遷移関係
- 各画面の主要要素（ヘッダー、フォーム、リスト、ボタンなど）
- モーダル・ドロワー・トーストなどの補助 UI

**機能リスト**
- 各画面で実現すること
- バリデーション要件
- エラー状態・空状態・ローディング状態

**優先度リスト**
- Must Have（必須）
- Should Have（重要）
- Nice to Have（余裕があれば）

### 4. ユーザーへの確認

要件をまとめた後、ユーザーに確認を求める。特に曖昧な点や判断が難しい箇所は明示的に質問する。

## 成果物

### 必須（Markdown に記録）

`docs/requirements/[機能名].md` に画面要件定義書を作成:

```markdown
# [機能名] 画面要件定義書

## 目的
[この画面で何を実現するか]

## 対象ユーザー
[誰が使うか]

## 画面構成
### [画面名]
- 役割:
- 主要要素:
- 遷移先:

## 機能リスト
| 機能 | 優先度 | 備考 |
|------|--------|------|
| ... | Must | ... |

## 非機能要件
- パフォーマンス:
- アクセシビリティ:
```

### 状況に応じて（Issue コメントに記録）

- 画面要否の判断根拠
- ユーザーストーリー連携マップ

## 次のステップ

- 画面が必要 → 「3. 静的画面の作成 (`dev-static-screen`)」への移行を提案
- 画面が不要 → 「4. 実装計画の立案 (`dev-implementation-plan`)」への移行を提案

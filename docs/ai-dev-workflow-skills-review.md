# AI駆動開発ワークフロー Skill 仕様確認と疑問点

`docs/ai-dev-workflow-skills.md` の内容を確認し、各 Skill に対する疑問点と実装前提を整理したメモです。

## 1. 現状確認 Skill
- 対象リポジトリ依存での「自動解析範囲」はどこまでか？
  - フロント/バック/infra/テストコードすべて or 主要構成 only
  - **回答:**すべて。「xxxを実装して」で起動するイメージ。
- 未定義項目を「ユーザー問い合わせ」するフロー実装方法
  - GitHub issue draft / Chatbot prompt / CLI対話 など
  - **回答:**issueにコメント

## 2. 要件定義 Skill
- 「画面が必要か」判定の自動判定ロジックは何基準か
  - UI/UXメタデータ or ドメインルール
  - **回答:**実装してほしいものから判別できませんか？
- 入力データ（ユーザーストーリー等）の収集形式
  - PR本文/issue形式/既存仕様書/設計文書
  - **回答:**すべて
- API定義は行わずの境界が曖昧（設計に反映する程度まで可か）
  - **回答:**静的画面の開発に不要なものはこのステップの範囲外とする。画面を軸にユーザーと認識合わせをしていきたいです。

## 3. 静的画面の作成 Skill
- UIモック生成はコード出力（React, Vue, HTML/CSS）かデザイン画像手順か
  - **回答:**フロントの技術スタックによる。これを1. 現状確認 Skillで確認が必須
- Storybook連携要件（既存プロジェクトに Storybook あるか確認）
  - **回答:**上記と同じ
- フィードバック収集仕組み：現場向けテンプレート or 自動仕組み
  - **回答:**静的ページを直接組み込めばよいのでは？ /profile/{id} にプロフィール画面を作成するのであればここに静的画面を作成する

## 4. 実装計画の立案 Skill
- DBスキーマ案の深さ（概念・論理・物理）と出力形式
  - **回答:**論理
- 既存設計との整合（既存 API のラップ/拡張か新規設計か）
  - **回答:**開発するものによるから定義できないのでは？新規画面だったら新規設計だし、既存画面の修正だったら拡張になると思います
- タスク分解の粒度（チケット1つ/サブタスク細分/CI階層）
  - **回答:**issueごと

## 5. テストコード作成 Skill
- 対応対象: 単体+統合+E2E。
- 実行環境（Jest/Cypress/Playwright）をどう決定？
  - **回答:**1の収集結果に合わせる
- カバレッジ測定および結果フィードバックの統合方法
  - **回答:**1の収集結果に合わせる

## 6. コード実装 Skill
- 逐次コミット候補の粒度基準（1機能1コミット or 1タスク1コミット）
  - **回答:**1タスク1コミット
- 単体テスト合格基準（合格＝PRマージ条件）
  - **回答:**合格＝PRマージ条件
- 既存CIフォーマット遵守と自動 lint/format 適用手順
  - **回答:**1. 現状確認 Skill の結果による

## 7. リファクタリング Skill
- 技術負債検出の定義（静的解析ルール/レビューフィードバック/実行結果）
  - **回答:**1. 現状確認 Skill　の結果による

## 8. PR作成とレビュー依頼 Skill
- PRテンプレート基準（最小必須フィールド）
  - **回答:** 1. 現状確認 Skill　の結果による
- レビューチェックリスト自動生成ルール
  - **回答:** 1. 現状確認 Skill　の結果による
- レビューコメント対応のトラッキング（issue + PR conversation）
  - **回答:** PR

## 追加: ハンドオフ/ジェネレーション連携 Skill
- 状態保存データ形式（JSON/YAML/DB）
  - **回答:** これはskillが分断されているからどうやって連携するかという意味ですか？であればissue。
- 次ステップ引継ぎの最低情報セット
  - **回答:**成果物の場所が分かる情報

## 横断的疑問点

### Skill 間の連携
- 各 Skill の出力が次の Skill の入力になる想定だが、受け渡し方法は？
  - ファイルパス参照 / Issue リンク / 明示的な引数指定 / ハンドオフ Skill 経由
  - **回答:**Issue か 各skillを続けて実行するのでチャットの終了結果で出す
- 前 Skill の成果物が未完了 or 省略された場合、次 Skill はどう振る舞うか（エラー/スキップ/補完）
  - **回答:**エラー

### Skill の発動トリガー
- 各 Skill をどう呼び出すか（ユーザーが明示的に `/skill-name` するのか、前 Skill 完了後に自動提案するのか）
  - **回答:**どちらとも
- シーケンシャル実行（前 Skill 完了を待つ）とスタンドアロン実行（任意の Skill を単独起動）の両方をサポートするか
  - **回答:**両方をサポート

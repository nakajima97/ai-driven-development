# Skills 一覧

このプロジェクトで利用可能なスキル（`/skill-name` で呼び出し）の一覧です。

## AI駆動開発ワークフロー

開発は以下の順序で進めます。各ステップは前のステップの完了を前提とします。

| ステップ | スキル名 | 役割 |
|--------|---------|------|
| 1 | `dev-status-check` | プロジェクトの技術スタック・依存関係・ディレクトリ構造・コーディング規約を収集 |
| 2 | `dev-requirements` | 画面要件を構造化して整理 |
| 3 | `dev-static-screen` | UIモック・静的ページを生成して画面要件を固める |
| 4 | `dev-implementation-plan` | 実装に必要なAPI・DB・アーキテクチャ計画を設計 |
| 5 | `dev-test-creation` | 画面を壊さないためのテストを生成 |
| 6 | `dev-code-implementation` | テストをグリーンにするためのコードを実装 |
| 7 | `dev-refactoring` | 実装済みコードの品質改善・パフォーマンス向上・ドキュメント整備 |
| 8 | `dev-pr-review` | PRを作成してレビュー依頼 |

---

## 各スキルの詳細

### 1. dev-status-check

**役割**: AI駆動開発ワークフローの最初のステップ。プロジェクトの現状を把握する。

**使う場面**:
- 「現状確認して」
- 「技術スタックを調べて」
- 「このプロジェクトの構成を把握して」
- 新しい開発タスクを始める前（必須）

**成果物**: 現状確認レポート（技術スタック・ディレクトリ構造・コーディング規約・未定義項目）

---

### 2. dev-requirements

**役割**: 画面要件を構造化して整理する。API定義は行わず、画面の目的・構成・機能にフォーカス。

**前提**: `dev-status-check` の完了

**使う場面**:
- 「要件定義して」
- 「画面の仕様を決めて」
- 「何を作るか整理して」
- 「ユーザーストーリーをまとめて」

**成果物**: `docs/requirements/[機能名].md`（画面要件定義書）

---

### 3. dev-static-screen

**役割**: 画面要件定義をもとに、実際のルートに静的ページを作成して要件を詰める。バックエンド連携なし。

**前提**: `dev-requirements` の完了

**使う場面**:
- 「静的画面を作って」
- 「UIモックを作成して」
- 「画面のプロトタイプを見せて」
- 「デザインをコードで確認したい」

**成果物**: 静的画面コンポーネント群・モックデータファイル

---

### 4. dev-implementation-plan

**役割**: 静的画面で確認済みの画面仕様をもとに、バックエンド機能・API・DBスキーマ・タスク分解を設計する。

**前提**: `dev-static-screen` の完了（または `dev-requirements` で画面不要と判断した場合）

**使う場面**:
- 「実装計画を立てて」
- 「APIを設計して」
- 「DBスキーマを考えて」
- 「バックエンドの設計をして」

**成果物**: `docs/implementation-plan/[機能名].md`（API定義・DBスキーマ・タスク分解リスト）

---

### 5. dev-test-creation

**役割**: 実装計画・画面仕様をもとに、画面とロジックを守るためのテストを作成する（TDDアプローチ）。

**前提**: `dev-implementation-plan` の完了

**使う場面**:
- 「テストを書いて」
- 「テストケースを作成して」
- 「ユニットテストを追加して」
- 「E2Eテストを書いて」

**成果物**: `*.test.ts` / `*.spec.ts` ファイル群・E2Eテストスペック

---

### 6. dev-code-implementation

**役割**: 実装計画・テスト仕様・静的画面をもとに、テストをパスするコードを実装する。1タスク1コミット単位で進める。

**前提**: `dev-test-creation` の完了

**使う場面**:
- 「実装して」
- 「コードを書いて」
- 「テストを通して」
- 「機能を追加して」

**成果物**: 実装差分パッチ・コード変更（リポジトリにコミット）

---

### 7. dev-refactoring

**役割**: 完了したコードの品質を向上させる。スコープは**今回の変更箇所のみ**。テストグリーンを維持しながら改善する。

**前提**: `dev-code-implementation` の完了・テストが全てグリーン

**使う場面**:
- 「リファクタリングして」
- 「コードをきれいにして」
- 「技術負債を解消して」
- 「パフォーマンスを改善して」

**成果物**: リファクタリング提案書（Issue コメント）・コード改善差分

---

### 8. dev-pr-review

**役割**: リファクタリングまで完了した成果をPRとして共有し、品質保証を進める。

**前提**: `dev-refactoring` の完了

**使う場面**:
- 「PRを作って」
- 「プルリクエストを作成して」
- 「レビュー依頼して」
- 「ブランチをpushして」

**成果物**: PRリンク・レビュー依頼メッセージ

---

## ユーティリティスキル

### create-issue

**役割**: GitHubのfeature_request issueを作成する。会話のコンテキストから内容を自動推論する。

**使う場面**:
- 「issueを作って」
- 「機能要望をissueにして」
- 「GitHubに起票して」
- 「今話した内容をissueにして」

**成果物**: GitHub issue URL

---

## ワークフロー図

```
開発タスク開始
     │
     ▼
1. dev-status-check（必須・毎回実行）
     │
     ▼
2. dev-requirements
     │
     ├─ 画面あり ──▶ 3. dev-static-screen ──▶ 4. dev-implementation-plan
     │
     └─ 画面なし ──────────────────────────▶ 4. dev-implementation-plan
                                                      │
                                                      ▼
                                             5. dev-test-creation
                                                      │
                                                      ▼
                                             6. dev-code-implementation
                                                      │
                                                      ▼
                                             7. dev-refactoring
                                                      │
                                                      ▼
                                             8. dev-pr-review
                                                      │
                                                      ▼
                                                   完了
```

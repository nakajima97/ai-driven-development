# AI駆動開発ワークフローのたたき台
## 前提
- Web開発を前提とする
- ドキュメントではなく実際のコードを正とする
  - ドキュメントはあくまで補助
- AIに与える以下の情報を事前準備する
  - 技術スタック・バージョン
  - プロジェクト構造・ディレクトリ構成
  - 命名規則・コーディング規約
  - 既存コードのパターン・慣例
  - 使用ライブラリ・ツール
- 技術スタックは以下を前提とする
  - フロント
    - React※メタフレームワーク（Next.JS、React Router等）は関係ない
    - Storybook
    - Vitest
  - バックエンド
    - 特になし（どのフレームワークを使ってもOpenAPI定義を作成するのは可能なので）

## コンセプト
- アウトプット駆動開発　という開発手法を新たに定義
- アウトプットをまず最初に確定させてそれを中心に機能を開発していく
  - アウトプット＝実装前に確定させるシステムの仕様・契約
  - 該当するもの
    - 静的画面（Storybook）
    - APIリクエスト・レスポンスのOpenAPI定義
    - ログの形式・出力条件
    - 非同期メッセージ・イベント定義（該当する場合）
    - DBスキーマ（マイグレーション定義）
- アプリのテストコード=仕様書とする
  - ドキュメントは理解を簡単にするための補助とする

## 手順
1. issue作成
   1. AIまたは人間がissueを作成する
2. 要件定義
   1. AIがissueを確認
      1. 機能要件・制約の不明点があれば人間に確認する
   2. AIが既存コードベースの調査を行う
   3. AIがアウトプットを生成する
   4. 人間がアウトプットを確認する
   5. AIと人間が相談してアウトプットを確定させる
3. 実装準備
   1. AIが2-2で決めたアウトプットを壊さないためのテストを箇条書きで書く
   2. 人間が確認する
   3. AIと人間による合意が取れたらAIが3-1で決めたテストを実装する
4. 実装
   1. AIが3-1の補助となるテストを記載する
   2. AIによる実装
   3. AIによるセルフレビュー
      1. セルフレビュー結果はAIが人間の承認を得ずに修正する
      2. 2回の修正で直せなかった場合は人間に修正できなかった旨を報告する

## 実現方針
- 手順の1-4がそれぞれskillとして存在する
- AIが行う各タスクをagentとして実装する
- ドキュメント更新が必要であれば更新する
  - 回す中で生成されるドキュメント
    - 画面一覧
    - 画面遷移図
    - ER図（DB構造を瞬時に把握するため）
    - API一覧
  - このドキュメントは要件定義で生成される
